# 驻波守卫融合策略修改说明

> 修改日期：2026-03-09  
> 涉及文件：`PhaseRangeFinder.kt`、`FusedProximityDetector.kt`

---

## 一、背景与问题

### 问题现象

在 0~5cm 的 VERY_NEAR 状态下，当手非常缓慢地移动时，会周期性遇到**声学驻波波节**。在驻波波节处，麦克风采集到的反射信号幅度会急剧下降，导致能量比从 >阈值 骤降到约 **0.88** 甚至更低，触发系统判断"手已离开"。

但实际上手**没有移动**，只是恰好停在了驻波的波节位置。手继续缓慢移动后，反射能量会重新上升，系统又判断"手回来了"，造成**状态抖动**和**2cm 附近停滞**的感知体验。

### 根本原因

原始决策逻辑**完全依赖能量比**来判断是否降级（VERY_NEAR → PENDING/FAR），没有对"能量下降是由驻波引起还是手真的在离开"做任何区分。

### 关键规律（用于破解）

| 情形 | 能量比变化 | LLAP Δd（相位速度） |
|------|-----------|-------------------|
| 手正常靠近 | ↑ 增大 | < 0，绝对值增大 |
| 手驻留在驻波波节（手未动） | ↓ 急剧下降 | **≈ 0（手没动）** |
| 手正常远离 | ↓ 下降 | > 0，绝对值增大 |

**破解点**：驻波让**能量波动**，但不改变手的实际位置，而 LLAP（相位法）测量的是**位移的导数（速度）**，对驻波不敏感，天然可以作为能量法降级的"仲裁者"。

## 二、PhaseRangeFinder.kt 改动

### 2.1 新增字段：速度窗口

```kotlin
// 速度窗口：保存最近5帧的 Δd (mm)，用于驻波检测
private val VELOCITY_WINDOW_SIZE = 5
private val recentDeltas = FloatArray(VELOCITY_WINDOW_SIZE)
private var velocityWindowPos = 0
private var velocityWindowFilled = false
```

**目的**：维护一个近5帧的环形缓冲，平滑单帧噪声，得到稳定的近期速度估计。

### 2.2 processFrame：每帧写入速度窗口

**修改前**：`processFrame` 只返回本帧 Δd，外部无法感知历史速度趋势。

**修改后**：在 `processFrame` 返回前，将 Δd 写入环形缓冲：

```kotlin
// 更新速度窗口
recentDeltas[velocityWindowPos] = distance
velocityWindowPos = (velocityWindowPos + 1) % VELOCITY_WINDOW_SIZE
if (velocityWindowPos == 0) velocityWindowFilled = true
```

### 2.3 新增公开方法

#### `getRecentVelocity(): Float`
返回近5帧 Δd 的均值（mm/帧）。
- **正值**：手在远离
- **负值**：手在靠近
- **≈ 0**：手静止（驻波候选）

```kotlin
fun getRecentVelocity(): Float {
    val count = if (velocityWindowFilled) VELOCITY_WINDOW_SIZE else velocityWindowPos
    if (count == 0) return 0f
    return recentDeltas.take(count).sum() / count
}
```

#### `isHandStationary(stationaryThreshold: Float = 0.3f): Boolean`
便捷方法，判断手是否静止（速度绝对值低于阈值）。

```kotlin
fun isHandStationary(stationaryThreshold: Float = 0.3f): Boolean {
    return kotlin.math.abs(getRecentVelocity()) < stationaryThreshold
}
```

### 2.4 reset() 同步清空速度窗口

```kotlin
// 重置速度窗口
recentDeltas.fill(0f)
velocityWindowPos = 0
velocityWindowFilled = false
```

**目的**：防止重置后残留历史速度数据影响下次会话的判断。

## 三、FusedProximityDetector.kt 改动

### 3.1 新增常量

```kotlin
// 驻波守卫：LLAP速度绝对值 < 此值时认为手静止（驻波）
private const val STATIONARY_VELOCITY_THRESHOLD = 0.3f  // mm/帧

// 驻波守卫：远离速度需大于此值才允许降级
private const val LEAVING_VELOCITY_THRESHOLD = 0.1f  // mm/帧（正值=远离）
```

### 3.2 新增字段：最新相位速度

```kotlin
// 驻波守卫：最新一帧的 LLAP 速度（由相位处理循环更新）
private var latestPhaseVelocity = 0f
```

### 3.3 相位处理循环：更新速度字段

```kotlin
val distanceChange = phaseRangeFinder.processFrame(phaseBuffer)
// 记录当前帧速度（processFrame 内已更新速度窗口，这里取最新均值）
latestPhaseVelocity = phaseRangeFinder.getRecentVelocity()
```

### 3.4 驻波期间禁止重置 LLAP 累积值

**修改前**：只要能量比接近阈值，就无条件重置 `relativeDistance`：

```kotlin
// 原代码（有问题）
if (energyRatio >= dynamicThreshold && abs(energyRatio - dynamicThreshold) < 0.05f) {
    if (relativeDistance != 0f) {
        onRelativeDistanceReset?.invoke(relativeDistance)
        relativeDistance = 0f
    }
}
```

**修改后**：加入驻波守卫，驻波期间禁止重置：

```kotlin
val isStandingWave = currentState == ProximityState.VERY_NEAR
        && energyRatio < dynamicThreshold
        && phaseRangeFinder.isHandStationary(STATIONARY_VELOCITY_THRESHOLD)

if (energyRatio >= dynamicThreshold && abs(energyRatio - dynamicThreshold) < 0.05f) {
    if (relativeDistance != 0f && !isStandingWave) {  // 驻波期间不重置
        onRelativeDistanceReset?.invoke(relativeDistance)
        relativeDistance = 0f
    }
}
```

**原因**：如果此时处于驻波状态，`relativeDistance` 保存的是手的历史位移信息，不应丢弃，后续手继续移动时还需要它。

## 四、状态决策逻辑对比

### 4.1 原始决策（5条规则）

```
determineState(energyRatio, relativeDistance, lowEnergyDuration):

1. energyRatio >= dynamicThreshold
       → VERY_NEAR

2. energyRatio < dynamicThreshold AND relativeDistance > 50mm
       → FAR

3. lowEnergyDuration >= 1000ms
       → FAR

4. else
       → PENDING
```

**问题**：规则2和规则3完全由能量主导，驻波导致的能量下降会被误判为手在离开，触发降级。

---

### 4.2 新决策（8条规则，含驻波守卫）

```
determineState(energyRatio, relativeDistance, lowEnergyDuration):

辅助变量：
  isStandingWave  = (currentState == VERY_NEAR)
                  AND (energyRatio < dynamicThreshold)
                  AND (|LLAP速度| < 0.3mm/帧)         ← 手静止

  isLeavingByPhase = (LLAP速度 > 0.1mm/帧)            ← 确认远离


规则优先级（从高到低）：

1. energyRatio >= dynamicThreshold
       → VERY_NEAR
       （能量充足，无争议靠近）

2. isStandingWave == true
       → VERY_NEAR（抑制降级）
       （能量虽下降，但 LLAP 证明手没动 → 驻波，不降级）

3. energyRatio < dynamicThreshold
   AND relativeDistance > 50mm
   AND isLeavingByPhase == true
       → FAR
       （能量下降 + 距离够大 + LLAP 三重确认 → 确实在离开）

4. energyRatio < dynamicThreshold
   AND relativeDistance > 50mm
   AND isLeavingByPhase == false
       → PENDING
       （能量下降 + 距离够大，但 LLAP 无法确认 → 先保持待定，不贸然降级）

5. lowEnergyDuration >= 1000ms
   AND isLeavingByPhase == true
       → FAR
       （低能量超时 + LLAP 确认远离 → 降级）

6. lowEnergyDuration >= 1000ms
   AND isHandStationary == true
   AND currentState == VERY_NEAR
       → VERY_NEAR（抑制降级）
       （低能量超时，但 LLAP 证明手没动 → 驻波持续，继续保持靠近）

7. lowEnergyDuration >= 1000ms
       → FAR
       （低能量超时，且无驻波守卫触发 → 正常降级）

8. else
       → PENDING
```

---

### 4.3 新增决策的核心思路图

```
能量下降触发降级检查
        │
        ▼
  查询 LLAP 近期速度
        │
   ┌────┴────────────────┐
   │                     │
速度 ≈ 0              速度 > 阈值
（手静止）            （手在远离）
   │                     │
   ▼                     ▼
① 驻波守卫             ② 正常降级
  抑制降级              允许 FAR
  保持 VERY_NEAR
```

## 五、阈值参数说明

| 参数 | 值 | 含义 |
|------|-----|------|
| `STATIONARY_VELOCITY_THRESHOLD` | 0.3 mm/帧 | LLAP 速度绝对值低于此值认为手静止 |
| `LEAVING_VELOCITY_THRESHOLD` | 0.1 mm/帧 | LLAP 速度高于此值才确认手在远离 |
| `VELOCITY_WINDOW_SIZE` | 5 帧 | 速度均值窗口大小（约 100ms，960样本/帧） |

> **调优建议**：
> - 若出现"驻波守卫误触发"（手确实在离开但未降级），可适当**降低** `STATIONARY_VELOCITY_THRESHOLD`（如 0.15）
> - 若驻波仍然导致闪烁，可适当**提高** `STATIONARY_VELOCITY_THRESHOLD`（如 0.5）或扩大 `VELOCITY_WINDOW_SIZE`（如 8）

## 六、resetRelativeDistance 联动修复

**修改前**：每次重置只清零 `relativeDistance` 和 `phaseRangeFinder` 内部状态。

**修改后**：同时清零 `latestPhaseVelocity`，确保状态一致：

```kotlin
private fun resetRelativeDistance(reason: String) {
    relativeDistance = 0f
    latestPhaseVelocity = 0f   // 新增：清零速度，避免残留影响
    phaseRangeFinder.reset()
}
```

**原因**：如果不清零，重置后的第一帧可能使用上一次会话的速度值，导致错误的守卫判断。

## 七、预期效果

| 场景 | 修改前 | 修改后 |
|------|--------|--------|
| 手靠近后停在驻波节点 | 能量骤降 → 状态抖动 → 显示离开 | LLAP速度≈0 → 驻波守卫 → 保持 VERY_NEAR |
| 手持续缓慢靠近经过驻波 | 2cm处停滞，状态反复切换 | 穿越驻波时保持靠近状态，平滑通过 |
| 手真实快速离开 | 正常降级 | 同样正常降级（LLAP确认速度>0.1） |
| 手真实缓慢离开 | 正常降级 | 可能短暂进入PENDING，但不会误判为驻波 |
